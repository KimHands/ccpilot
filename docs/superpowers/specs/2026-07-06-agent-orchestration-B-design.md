# B — 특화 서브에이전트 오케스트레이션 설계 문서

- 작성일: 2026-07-06
- 상태: 설계 확정 대기 (코덱스 리뷰 반영, 축소 v1)
- 선행: A(프로젝트 프리셋 워크플로우) — 완료·설치됨
- 범위: **B v1 — 이미 로드된 능력 사이의 특화 라우팅 + 종합**

## 1. 목적

구체 작업("이 코드 보안 점검해줘")에 대해, **이미 로드된** 플러그인 에이전트·스킬 중 최적을 골라 **독립 컨텍스트로 디스패치**하고, 여러 결과를 **구조화 종합**해 메인 세션을 가볍게 유지한다. 원본 에이전트·스킬은 **일절 수정하지 않는다**(저하 0의 유일한 보장).

## 2. 비목표 (v1 제외 · 후속)

- **세션 중 플러그인 hot-enable/disable** — Claude Code는 세션 시작 시 로드. 불가. (A/세션 재시작 담당)
- **intent→preset 자동 선택** — A의 `/project-init` 확장으로 분리.
- **persistent `agent-catalog.json`** — v1은 live 조회. 반복 라우팅 모호성이 실증되면 캐시로 도입.
- **범용 무한 라우팅** — v1은 2개 도메인(보안·테스트) + 미지 작업 폴백 규칙.
- **서브에이전트의 중첩 디스패치** — 미검증. v1 의존 금지.

## 3. 검증된 Claude Code 메커니즘 (전제)

- 에이전트 정의 = `~/.claude/agents/*.md`(user) / `<proj>/.claude/agents/*.md`(project). frontmatter: `name`, `description`, `tools`(허용목록), `model`, 본문 시스템 프롬프트. 플러그인은 자체 에이전트를 캐시에 제공.
- **서브에이전트는 메인 Claude가 Task/Agent 도구로 디스패치.** 관측된 에이전트 정의엔 Task/Agent 도구가 없음 → **서브에이전트는 다른 서브에이전트를 못 띄움**(중첩 불가). 스킬은 서브에이전트도 호출 가능.
- 플러그인은 세션 시작 시 로드.

## 4. 아키텍처 & 파일

```
~/.claude/
├── agents/
│   ├── security-auditor.md    # 얇은 종합 전문가 (스킬 호출 + 구조화 종합)
│   └── test-engineer.md       # 얇은 종합 전문가
├── commands/
│   └── route.md               # /route <작업> — 디스패치 "계획" 생성 (메인이 실행)
└── route-rules.json           # 작업→선호 에이전트·우선순위 규칙 (소량)
```
설치는 A의 `install.sh`를 확장해 함께 배포(§12).

**동작 흐름:** 사용자가 작업을 말함 → 메인 Claude가 `/route <작업>` 실행 → route.md가 `route-rules.json` + **live 에이전트/스킬 디렉토리 조회**로 **디스패치 계획** 생성 → 메인 Claude가 계획대로 Task 디스패치(병렬 가능) → 각 결과(구조화)를 메인이 **종합**.

## 5. 전문가 에이전트 설계 (얇게)

각 전문가는 **서브에이전트**이며, **스킬을 직접 호출**해 작업하고 **구조화 리포트**를 반환한다. 여러 *에이전트* 조합은 전문가 내부가 아니라 `/route` 계획(메인 레벨)이 담당한다.

**공통 규칙(코덱스 #1·#6):**
- 스킬·원본 플러그인 에이전트 파일을 **수정하지 않는다**.
- **고유 종합 가치**가 있을 때만 존재. 원본 단일 에이전트로 충분하면 `/route`가 원본을 우선 디스패치.
- 얇고 선언적: "무엇을 종합하는가 / 어떤 스킬·에이전트를 쓰나 / 무엇을 중복하지 말 것 / 스킬 없을 때 폴백".
- 이름·설명의 권위는 live 조회. 하드코딩된 절차 복제 금지.

**`security-auditor.md`** — `tools: Bash, Read, Glob, Grep`, `model: sonnet`.
- 쓰는 스킬: kesekit(KISA 점검), static-analysis(`/semgrep`·`/codeql`), security-guidance 가이드.
- 고유 가치: 세 관점(규제·규칙기반 정적분석·실시간 패턴)의 결과를 **하나로 중복제거·우선순위화**하고 이견 보존.
- 폴백: 스킬 미설치 시 해당 관점 생략하고 리포트에 명시.

**`test-engineer.md`** — `tools: Bash, Read, Write, Edit, Glob`, `model: sonnet`.
- 쓰는 스킬: test-writer-fixer, 감지 언어 LSP, 테스트 러너(pytest/jest 등).
- 고유 가치: 테스트 작성→실행→실패분석→수정 루프를 한 격리 컨텍스트에서 돌리고 커버리지 갭을 종합.
- 폴백: 러너/LSP 없으면 명시.

## 6. `/route` — 디스패치 계획 명령 (코덱스 #2)

`/route <작업 설명>` 은 **자율 실행자가 아니라 계획 생성기**다. route.md(프롬프트/제어 텍스트)가 메인 Claude에게:

1. `route-rules.json` 로드 + **live 조회**: `~/.claude/agents/*.md`, `<proj>/.claude/agents/*.md`, 플러그인 캐시의 `*/agents/*.md`, 사용 가능 스킬 목록을 읽어 **현재 실제 존재하는** 에이전트/스킬 파악.
2. 작업을 규칙에 매칭 → **디스패치 계획** 산출:
   - 선택된 에이전트/스킬 + **각 선택 이유**
   - 병렬 안전 여부
   - 각 디스패치의 **정확한 task 프롬프트**
   - 기대 **출력 계약**(§10)
   - **종합 체크리스트**
   - 건너뛸 항목과 이유
3. 메인 Claude가 계획대로 Task 디스패치 후 §10 규칙으로 종합.

**우선순위 규칙(코덱스 #5):** 광범위 보안 → `security-auditor`; 정밀 Semgrep만 → 원본 `semgrep-scanner`; 테스트 → `test-engineer`; **미지 작업 → 원본 플러그인 에이전트 우선**(큐레이티드 래퍼는 문서화된 통합 이점이 있을 때만).

## 7. `route-rules.json` + live 조회 + 신선도 (코덱스 #4)

v1은 **persistent 카탈로그를 만들지 않는다.** 대신:
- `route-rules.json`(소량): `{ "rules": [ {"match": ["보안","security","취약"], "prefer": "security-auditor", "narrow": {"semgrep만": "semgrep-scanner@static-analysis"}}, {"match": ["테스트","test","커버리지"], "prefer":"test-engineer"} ], "fallback": "prefer-original-plugin-agents" }`
- **live 조회로 항상 최신**: `/route` 실행 시 에이전트 디렉토리를 그때 읽으므로 드리프트 없음. 규칙이 가리키는 에이전트가 실제로 없으면 계획에서 제외하고 사유 표기.
- 규칙 파일은 사용자가 자유 편집(새 도메인 추가).

## 8. 저하 0 원칙 (테스트 가능하게 재정의, 코덱스 #1)

"무조건 향상"은 보장 대상이 아니라 **설계 불변식**으로 표현:
- **INV-1:** B는 어떤 스킬·플러그인 에이전트 파일도 수정/삭제하지 않는다.
- **INV-2:** 작업이 원본 플러그인 에이전트와 직접 매치되면 `/route`는 그 원본을 우선 디스패치(재구현 금지).
- **INV-3:** 종합은 개별 결과를 **버리지 않고** 이견을 보존한다.
향상은 (a)올바른 선택 (b)병렬 (c)교차-플러그인 종합에서만 나온다. 테스트: INV-1은 파일 해시 불변, INV-2/3은 계획·종합 출력 검증.

## 9. A/B 경계 (코덱스 #8)

- A: 프리셋·단계 상태 소유. B: **단계 상태 소유 안 함.**
- A 플레이북의 review/implement 단계가 "`/route`로 위임"을 안내할 수 있음(소프트 참조).
- **B는 A 없이도 동작** — live 카탈로그와 현재 작업만으로.

## 10. 구조화 출력 & 종합 규칙 (코덱스 #7)

디스패치되는 모든 에이전트는 아래를 반환해야 한다:
```
- findings: [...]
- evidence: 파일/라인/명령 출력
- inspected: 조사한 파일·명령
- confidence: high|med|low
- unresolved: 미해결 질문
- next: 권장 다음 행동
```
**종합 규칙:** 이견 보존 + 각 주요 주장에 **출처 에이전트 표기** + 과장 금지(예: "도구가 X 보고" ≠ "X는 확실히 익스플로잇 가능", 전문가가 명시 결론 냈을 때만 승격).

## 11. 동적 플러그인 한계 (코덱스 #3, 명시)

- B v1은 **이미 로드된** 에이전트/스킬 사이에서만 라우팅.
- hot-enable/disable 안 함.
- 작업이 현재 프리셋 밖 능력을 요구하면 `/route`는 계획 대신 **"이 세션엔 없음 — 프리셋 켜고 재시작(/project-activate)"** 을 보고.

## 12. 설치 & 배포

- A의 `install.sh`를 확장: `agents/*.md` → `~/.claude/agents/`, `commands/route.md` → `~/.claude/commands/`, `route-rules.json` → `~/.claude/route-rules.json` 복사(기존 pptx 에이전트 등 **덮어쓰지 않음** — 동명 없음 확인).
- `~/.claude/agents/`의 기존 사용자 에이전트는 보존.

## 13. 엣지 케이스

- **규칙이 가리키는 에이전트 부재** → 계획에서 제외 + 사유.
- **작업이 여러 도메인** → 계획에 복수 전문가(병렬 안전 표기).
- **스킬 일부 미설치** → 전문가가 폴백·명시.
- **원본과 큐레이티드가 둘 다 매치** → §6 우선순위로 하나 선택(광범위=큐레이티드, 정밀=원본).
- **프리셋 밖 능력 요구** → §11 보고.

## 14. 테스트 전략

- `route-rules.json` 로드·검증(스키마).
- live 조회: 임시 에이전트 디렉토리에 더미 `*.md`를 두고, 규칙 매칭이 존재하는 것만 계획에 넣는지 / 부재 시 제외하는지.
- 계획 출력 형식(선택·이유·병렬·프롬프트·출력계약·종합체크) 존재 검증.
- INV-1: security-auditor/test-engineer 프롬프트에 "원본 스킬·에이전트 파일 수정 금지" 불변식 문구 포함 검증.
- 프리셋 밖 능력 요구 시 "재시작 안내" 경로.
- (전문가 에이전트 자체의 종합 품질은 실사용 관측 — 자동 단위테스트 대상 아님.)

## 15. 향후 (v2+)

- persistent `agent-catalog.json`(신선도 해시·타임스탬프) — 반복 라우팅 모호성 실증 시.
- intent→preset (A 확장): 대화 시작 시 "X 할거야" → 프리셋 추천.
- 전문가 추가(perf-reviewer, docs-writer 등).
- A 플레이북에 `/route` 참조 자동 삽입.
